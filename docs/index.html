Product Brief

Overview

The AES-256 Cryptographic IP Core is a register-controlled hardware accelerator for SoC, FPGA, and ASIC designs. The IP exposes a compact AXI4-Lite control plane and internally routes 128-bit blocks through AES-256 key expansion, encryption/decryption datapaths, mode engines, status reporting, and interrupt generation.

Microarchitecture

Architecture

The wrapper separates software-visible AXI4-Lite register handling from the AES core data plane. Software programs key, IV, AAD, input blocks, mode, direction, and interrupt enables; the core performs key expansion and selected mode processing.

Control Plane

aes256_axi4_interface decodes AXI4-Lite transfers, stores software-programmed registers, issues one-block start commands, captures output, and manages interrupt state.

Data Plane

aes256_top_core expands the 256-bit key, selects ECB/CBC/CTR/CFB/OFB/GCM engines, runs the 14-round datapath, and returns valid output blocks or GCM tags.

Quick Start

Programming Model

Firmware operates the IP as a one-block-at-a-time register accelerator. Load key material first, wait for key expansion, then issue block commands.

  1. Write CTRL with START=0, selected ENC_DEC, and selected MODE.
  2. Write KEY0..KEY7 with the 256-bit AES key.
  3. Poll STATUS.KEY_READY until the expanded key is available.
  4. For IV-based modes, write IV0..IV3.
  5. Write DIN0..DIN3 and configure DIN_CTRL.din_last.
  6. Write CTRL.START=1 to issue the block command.
  7. Poll STATUS.DONE or wait for o_aes_irq.
  8. Read DOUT0..DOUT3; for GCM, also read TAG0..TAG3 when tag status is pending.
  9. Clear pending interrupt status by writing one to IRQ_STAT bits.
wr32(AES_CTRL, MODE_CBC | AES_ENCRYPT);for (int i = 0; i < 8; i++) wr32(AES_KEY0 + 4*i, key[i]);while ((rd32(AES_STATUS) & STATUS_KEY_READY) == 0) { }for (int i = 0; i < 4; i++) wr32(AES_IV0 + 4*i, iv[i]);wr32(AES_DIN_CTRL, DIN_LAST);for (int i = 0; i < 4; i++) wr32(AES_DIN0 + 4*i, plaintext[i]);wr32(AES_CTRL, MODE_CBC | AES_ENCRYPT | CTRL_START);while ((rd32(AES_STATUS) & STATUS_DONE) == 0) { }for (int i = 0; i < 4; i++) ciphertext[i] = rd32(AES_DOUT0 + 4*i);wr32(AES_IRQ_STAT, IRQ_DOUT_VALID | IRQ_TAG_VALID);
WARNDo not modify mode, direction, key, IV, or data registers while the core is busy. Program every word of a multiword register group before asserting CTRL.START.
INFOFor ECB and CBC, ENC_DEC selects encryption or decryption. CTR, CFB, OFB, and GCM are stream-style transforms built around the encryption datapath.

Timing and Latency

Timing

The register wrapper and AES core are synchronous to i_aes_aclk. Valid strobes are level-based: w_core_key_valid stays high once all eight KEY words are written (write-mask = 0xFF), and w_core_din_valid stays high once all four DIN words complete (mask = 0xF). Both masks auto-clear after the core accepts the data. CTRL.START generates a rising-edge w_start_pulse that clears previous DONE status and re-arms output capture.

Cycle Budget Total single-block latency = 13 (key expand) + 1 (load) + 14 (rounds) + 1 (output) = 29 clock cycles from key write completion. Subsequent blocks on the same key skip key expansion and take 1 + 14 + 1 = 16 clock cycles.

1. Key Expansion Timing

After firmware writes all eight KEY registers, r_key_wr_mask reaches 0xFF and drives w_core_key_valid high (level). The key expansion FSM takes 13 cycles (r_step = 0 → 12). On completion, key_ready and rkey_valid assert; the write mask auto-clears.

2. Single-Block ECB Encrypt (Full Flow)

Complete flow from reset: write CTRL → write KEY → wait key expansion → write DIN → START pulse → DP_LOAD → 14 rounds → DP_OUTPUT → DONE. STATUS.BUSY = (r_busy && !key_ready) || (rkey_valid == 0).

3. Back-to-Back Block Pipeline

After key expansion completes, subsequent blocks skip key loading. The datapath can accept a new din_valid in the same cycle it outputs (DP_OUTPUT → DP_LOAD shortcut when din_valid && rkey_valid). Each additional block takes 16 cycles.

SoC Integration

Integration Notes

Reset / Clock

Use i_aes_aclk as the single AXI/core clock. Hold i_aes_aresetn low during reset and reinitialize software-visible registers after release.

AXI Access

Use 32-bit aligned AXI4-Lite accesses. Complete all words of key, IV, DIN, AAD, and control fields before asserting CTRL.START.

IRQ Handling

Enable desired events through IRQ_EN. Service DOUT and TAG registers, then clear IRQ_STAT with write-one-to-clear.

Security

Add system-level key zeroization, side-channel review, fault response policy, and authenticated-mode negative tests before production security deployment.

SECURITYThis documentation describes functional RTL behavior. Production certification requires threat modeling, implementation hardening, and expanded compliance vector testing.
